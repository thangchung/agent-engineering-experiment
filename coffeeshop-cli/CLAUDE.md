# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

**coffeeshop-cli** is a .NET/C# TUI (Terminal UI) CLI tool built with [Spectre.Console](https://spectreconsole.net/). It draws architectural inspiration from [Google Workspace CLI](https://github.com/googleworkspace/cli) — a tool that dynamically builds its command surface from discovery documents.

The CLI exposes:
- **Data models** — browse and inspect domain models (docs) in a tree/table TUI
- **Skills** — list and invoke agent skills (e.g., coffeeshop ordering from `agent-skills-coffeeshop`)
- **JSON submission** — submit hierarchical JSON payloads against data models
- **MCP server** — expose tools via Model Context Protocol in stdio transport mode

# Core Principles

### I. Minimal Dependencies

Every feature MUST use the smallest possible set of third-party libraries.
Prefer platform and framework built-ins over external packages.
Each dependency added to the project MUST be explicitly justified with a
rationale (capability not available in the framework, or would require
disproportionate effort to build independently).
Transitive dependencies count — prefer libraries with small dependency trees.

### III. Test-First (NON-NEGOTIABLE)

TDD is mandatory: tests MUST be written and reviewed before implementation code.
The Red-Green-Refactor cycle MUST be strictly followed:
1. Write a failing test that describes the desired behavior.
2. Obtain user/reviewer acknowledgement that the test correctly captures intent.
3. Write the minimum implementation to make the test pass.
4. Refactor while keeping tests green.
No PR may be merged with untested public surface area.

**Unit tests**: scope is a single class or function in isolation (mocked
dependencies). Framework: xUnit (backend) / Vitest + React Testing Library
(frontend).

**Integration tests** (MUST use .NET Aspire):
- All integration tests — for both the backend API and the frontend — MUST use
  `DistributedApplicationTestingBuilder` from the `Aspire.Hosting.Testing`
  NuGet package.
- The test AppHost starts the real application stack (backend, frontend, and
  any declared infrastructure resources) in a controlled test environment.
- Tests obtain typed `HttpClient` instances via `app.CreateHttpClient("<name>")`
  where `<name>` matches the resource name registered in the AppHost.
- Tests MUST wait for resources to reach `KnownResourceStates.Running` via
  `ResourceNotificationService.WaitForResourceAsync` before making assertions.
- Using `WebApplicationFactory<T>` for cross-service or full-stack integration
  tests is NOT permitted; it is only acceptable for intra-service unit-level
  HTTP tests that do not require other services or infrastructure.

### V. Observability

OpenTelemetry Protocol (OTLP) is the mandatory telemetry wire format for both
the backend and frontend. Signals MUST include traces, metrics, and logs.
The Aspire Dashboard (OTLP endpoint auto-exposed by the AppHost) is the
canonical local telemetry sink; `dotnet run --project apphost/` is the only
command needed to bring it up.

**Backend rules**:
- Structured logging (JSON) is REQUIRED for all request handling, service
  operations, and error paths.
- The backend MUST expose a `/healthz` endpoint returning service health.
- All unhandled exceptions MUST be caught at the API boundary, logged with a
  correlation ID, and returned as an RFC 9457 Problem Details response.
- OTel SDK configuration (SDK setup, OTLP exporter, ASP.NET Core & HTTP
  instrumentation) MUST be provided by the shared `ServiceDefaults` project
  (generated by the Aspire template). Manual per-service OTel SDK wiring that
  duplicates `ServiceDefaults` is NOT permitted.

### VI. Simplicity (YAGNI)

Features MUST solve the stated requirement — no speculative generalization.
Abstractions are introduced only when the same logic appears in three or more
independent places (Rule of Three).
Configuration MUST default to sensible values; do not expose options that have
no realistic use case in this project.

## Build & Run

```sh
# Restore and build
dotnet build

# Run the CLI
dotnet run -- <command> [options]

# Run a specific command example
dotnet run -- models list
dotnet run -- skills invoke coffeeshop-counter-service

# Run tests
dotnet test

# Run a single test
dotnet test --filter "FullyQualifiedName~TestClassName.TestMethodName"
```

## Architecture

### Command Tree (Spectre.Console.Cli pattern)

Commands follow a hierarchical `<noun> <verb>` pattern, similar to Google Workspace CLI:

```
coffeeshop-cli
├── models
│   ├── list          — List available data models
│   ├── show <name>   — Display model schema as tree/table
│   └── submit <name> — Submit JSON payload against a model
├── skills
│   ├── list          — List available agent skills
│   ├── show <name>   — Display skill details (from SKILL.md)
│   └── invoke <name> — Run a skill interactively
├── mcp
│   └── serve         — Start MCP stdio server exposing tools
└── docs
    └── browse        — Interactive TUI for browsing domain docs
```

### Key Patterns

**Command registration** — All commands inherit from `Command<TSettings>` or `AsyncCommand<TSettings>`. Settings classes use `[CommandArgument]` and `[CommandOption]` attributes. Commands are registered in `Program.cs` via `config.AddBranch(...)` / `config.AddCommand(...)`.

**Discovery-driven design** — Inspired by Google Workspace CLI, the tool should dynamically discover data models and skills from the filesystem or configuration rather than hardcoding them. Models are loaded from schema files; skills are loaded from `SKILL.md` manifests.

**MCP stdio transport** — The `mcp serve` command runs an MCP server over stdin/stdout, exposing the same capabilities (model queries, skill invocation) as tools that AI agents can call. This follows the [Model Context Protocol](https://modelcontextprotocol.io/) specification.

**JSON hierarchy** — When submitting data, the CLI accepts nested JSON that maps to hierarchical domain models (e.g., an order with line items). Validation happens against the discovered schema before submission.

### Relationship to Sibling Projects

- **agent-skills-coffeeshop** — Contains `SKILL.md` manifests and MCP server definitions for the coffeeshop domain. This CLI can discover and invoke those skills.
- **DotNetClaw** — The multi-channel agent runtime. This CLI is a complementary local-first interface to the same domain, not a channel for DotNetClaw.

## Conventions

- Use `Spectre.Console.AnsiConsole` for all TUI output (tables, trees, panels, status spinners) — never raw `Console.WriteLine`
- Commands that produce structured data should support `--json` flag for machine-readable output alongside the default TUI rendering
- Use `ITypeRegistrar` / `ITypeResolver` for DI integration with Spectre.Console.Cli
- Keep MCP tool definitions in sync with CLI commands — both should expose the same capabilities
- Follow the existing repo pattern: solution file at project root (`coffeeshop-cli.slnx`), source in a subfolder if using Aspire, otherwise flat
