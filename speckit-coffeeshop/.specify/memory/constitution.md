# CoffeeShop Constitution

## Core Principles

### I. Minimal Dependencies

Every feature MUST use the smallest possible set of third-party libraries.
Prefer platform and framework built-ins over external packages.
Each dependency added to the project MUST be explicitly justified with a
rationale (capability not available in the framework, or would require
disproportionate effort to build independently).
Transitive dependencies count — prefer libraries with small dependency trees.

### II. API-First Design

The backend MUST define all public contracts as typed Minimal API endpoints
before any frontend work begins on a feature.
Endpoints MUST use explicit request/response record types (no anonymous objects).
API routes MUST be versioned from day one (e.g., `/api/v1/...`).
The frontend MUST consume backend contracts through typed service clients;
inline `fetch` calls with untyped responses are not permitted.

### III. Test-First (NON-NEGOTIABLE)

TDD is mandatory: tests MUST be written and reviewed before implementation code.
The Red-Green-Refactor cycle MUST be strictly followed:
1. Write a failing test that describes the desired behavior.
2. Obtain user/reviewer acknowledgement that the test correctly captures intent.
3. Write the minimum implementation to make the test pass.
4. Refactor while keeping tests green.
No PR may be merged with untested public surface area.

**Unit tests**: scope is a single class or function in isolation (mocked
dependencies). Framework: xUnit (backend) / Vitest + React Testing Library
(frontend).

**Integration tests** (MUST use .NET Aspire):
- All integration tests — for both the backend API and the frontend — MUST use
  `DistributedApplicationTestingBuilder` from the `Aspire.Hosting.Testing`
  NuGet package.
- The test AppHost starts the real application stack (backend, frontend, and
  any declared infrastructure resources) in a controlled test environment.
- Tests obtain typed `HttpClient` instances via `app.CreateHttpClient("<name>")`
  where `<name>` matches the resource name registered in the AppHost.
- Tests MUST wait for resources to reach `KnownResourceStates.Running` via
  `ResourceNotificationService.WaitForResourceAsync` before making assertions.
- Using `WebApplicationFactory<T>` for cross-service or full-stack integration
  tests is NOT permitted; it is only acceptable for intra-service unit-level
  HTTP tests that do not require other services or infrastructure.

### IV. Component-Based UI

The frontend MUST be composed from shadcn UI primitives; custom components MUST
extend shadcn components rather than build from scratch.
Styling MUST use Tailwind CSS utility classes; hand-written CSS is only permitted
when a required visual behavior is demonstrably impossible with Tailwind alone,
and MUST be isolated in a CSS Module file co-located with the component.
Every interactive component MUST meet WCAG 2.1 AA accessibility requirements
(keyboard navigability, ARIA labels, sufficient color contrast).

### V. Observability

OpenTelemetry Protocol (OTLP) is the mandatory telemetry wire format for both
the backend and frontend. Signals MUST include traces, metrics, and logs.
The Aspire Dashboard (OTLP endpoint auto-exposed by the AppHost) is the
canonical local telemetry sink; `dotnet run --project apphost/` is the only
command needed to bring it up.

**Backend rules**:
- Structured logging (JSON) is REQUIRED for all request handling, service
  operations, and error paths.
- The backend MUST expose a `/healthz` endpoint returning service health.
- All unhandled exceptions MUST be caught at the API boundary, logged with a
  correlation ID, and returned as an RFC 9457 Problem Details response.
- OTel SDK configuration (SDK setup, OTLP exporter, ASP.NET Core & HTTP
  instrumentation) MUST be provided by the shared `ServiceDefaults` project
  (generated by the Aspire template). Manual per-service OTel SDK wiring that
  duplicates `ServiceDefaults` is NOT permitted.

**Frontend rules**:
- OTel MUST be initialized in `src/instrumentation.ts` (Next.js instrumentation
  hook) using `@vercel/otel` for server-side (Node.js runtime) traces and
  `@opentelemetry/sdk-web` with `@opentelemetry/exporter-trace-otlp-http` for
  browser-side traces.
- The OTLP endpoint URL MUST be read from the environment variable injected by
  the Aspire AppHost (`OTEL_EXPORTER_OTLP_ENDPOINT`); it MUST NOT be
  hard-coded.
- The frontend MUST surface user-facing error messages without leaking internal
  details; client errors MUST be recorded as OTel span events with enough
  context to correlate with backend traces.

### VI. Simplicity (YAGNI)

Features MUST solve the stated requirement — no speculative generalization.
Abstractions are introduced only when the same logic appears in three or more
independent places (Rule of Three).
Configuration MUST default to sensible values; do not expose options that have
no realistic use case in this project.

### VII. In-Memory State (NON-NEGOTIABLE)

The backend counter service is the single authoritative store for all
application state. No external database, cache, message broker, or file system
MAY be used as a persistence mechanism in this project.

- **All domain state** — customers, orders, menu items, and any per-session
  intent — MUST be held in process-local memory (e.g., `ConcurrentDictionary`,
  `List<T>`, or equivalent thread-safe collections registered as singletons in
  the DI container).
- **Restart resets state**: data loss on process restart is acceptable and
  expected behaviour. Code MUST NOT attempt to persist or reload state across
  restarts.
- **No ORM, no migrations**: Entity Framework Core, Dapper, and any other
  data-access library that targets a backing store are NOT permitted.
- **Seed data** (initial menu items, demo customers) MUST be loaded once at
  application startup from a static in-code definition, not from a file or
  external source.
- **Rationale**: Keeps the stack minimal (Constitution I), eliminates
  infrastructure dependencies from the local dev loop, and makes integration
  tests fast and dependency-free.

### Application Architecture

The backend is structured as **four discrete components** following clean architecture
principles. Three components run **in-process** within the same .NET host; one runs
**out-of-process** as a standalone MCP Server.

| Component | Process | Role |
|-----------|---------|------|
| **counter** | in-process | Primary orchestrating agent; owns order intake, customer identification, intent classification, and order creation flow. The single entry point for all frontend requests. |
| **barista** | in-process | Handles barista-side order fulfilment (beverage preparation steps). Receives work items from counter via the internal async queue. |
| **kitchen** | in-process | Handles kitchen-side order fulfilment (food preparation steps). Receives work items from counter via the internal async queue. |
| **product-catalog** | out-process (MCP Server) | Exposes the product catalog (item types, prices) as an MCP Server over HTTP/SSE. counter consumes it via the MCP protocol. |

**Communication rules**:
- `counter → barista` and `counter → kitchen`: **asynchronous, in-process C# queue**
  (e.g., `System.Threading.Channels.Channel<T>`). No external message broker is
  permitted (Constitution VII).
- `counter → product-catalog`: **MCP protocol over HTTP/SSE**. counter acts as an
  MCP client; product-catalog is an out-of-process MCP Server.
- `frontend → counter`: HTTP (Minimal API endpoints exposed by counter). Requests
  surface through CopilotKit's AG-UI protocol endpoint (`/api/copilotkit`).
- Direct inter-component calls that bypass the defined channels are NOT permitted.

**Order Fulfilment Flow** (counter as MAF orchestrator):

```
frontend ── POST order ──▶ counter (MAF agent, streaming via AG-UI)
                              │
                              ├─ classify each line item by category
                              │   └─ ◉ stream: "looking up product catalog…"
                              │
                              ├─ beverages only?  ──▶ enqueue to barista channel
                              │                    ◉ stream: "barista is preparing…"
                              │                         │
                              │                         ◀─ barista signals completion
                              │                    ◉ stream: "beverages ready"
                              │
                              ├─ food only?       ──▶ enqueue to kitchen channel
                              │                    ◉ stream: "kitchen is preparing…"
                              │                         │
                              │                         ◀─ kitchen signals completion
                              │                    ◉ stream: "food ready"
                              │
                              ├─ mixed (both)?    ──▶ enqueue to barista channel ─┐
                              │                    ──▶ enqueue to kitchen channel ─┤
                              │                    ◉ stream: "barista & kitchen preparing…"
                              │                         │                          │
                              │                         ◀─ await BOTH completions ─┘
                              │                            (Task.WhenAll / parallel)
                              │                    ◉ stream: per-worker completion as it arrives
                              │
                              └─ no items?        ──▶ return no-action response
                              │
                              ◀─ aggregate results
                              ◉ stream: "order ready — all items complete"
                              │
              frontend ◀── final order status ──┘
```

> **◉** marks indicate AG-UI status updates streamed to the frontend in
> real time while the counter awaits downstream processing.

1. **Receive** — counter receives the order request from the frontend via its
   Minimal API endpoint (surfaced through CopilotKit AG-UI).
2. **Classify** — the MAF orchestrator splits the order’s line items by
   category (`Beverages` → barista, `Food`/`Others` → kitchen) using the
   product-catalog data. A status update is streamed while the catalog lookup
   is in progress.
3. **Route** — four possible dispatch paths:
   - **Beverages only**: write work items to the barista `Channel<T>`.
   - **Food only**: write work items to the kitchen `Channel<T>`.
   - **Mixed**: write to **both** channels and `await Task.WhenAll(...)` on
     their completion signals (parallel, asynchronous).
   - **No items**: skip dispatch entirely; return a no-action acknowledgement.
4. **Stream progress** — while barista, kitchen, or product-catalog are
   processing, the counter MUST emit intermediate status updates to the
   frontend via the MAF streaming response (`RunStreamingAsync`) surfaced
   through the AG-UI protocol. Updates include:
   - **Dispatched**: which workers received items (e.g., “barista is
     preparing your latte”).
   - **Per-worker completion**: as each worker signals done, stream an
     update immediately (e.g., “beverages ready, waiting on kitchen…”).
   - **Catalog lookup**: when the product-catalog MCP call is in flight,
     stream a “looking up menu…” status.
5. **Complete** — barista and/or kitchen each write a completion signal back
   to the counter via a dedicated response `Channel<T>`. Counter aggregates
   the results once all expected signals arrive.
6. **Notify** — counter updates the order status (e.g., `preparing → ready`)
   in the in-memory store and streams the final status to the frontend,
   ending the AG-UI response stream.

**Rules**:
- The counter MUST NOT call barista or kitchen directly (no method calls);
  all dispatch goes through `Channel<T>` (Constitution VII).
- Mixed-order dispatch MUST be parallel (`Task.WhenAll`), never sequential.
- If either barista or kitchen fails during a mixed order, the counter MUST
  still await the other’s completion before responding with a partial-failure
  status.
- The no-action path MUST return immediately without writing to any channel.
- The counter MUST use `RunStreamingAsync` (MAF streaming) so that AG-UI
  status updates are pushed to the frontend in real time; the non-streaming
  `RunAsync` path is NOT permitted for order fulfilment.
- Status updates MUST be emitted at each processing boundary:
  (a) catalog lookup started, (b) items dispatched to worker(s),
  (c) each individual worker completion, (d) final aggregated result.
- The frontend receives these updates via CopilotKit’s AG-UI streaming
  protocol and MUST render them as progressive status indicators
  (e.g., inline messages in the chat sidebar or a status badge on the
  order card).

**Clean Architecture layers** (applied to each in-process component):
```
Domain/       # Entities, value objects, domain events (no external dependencies)
Application/  # Use cases, interfaces, service contracts
Infrastructure/ # In-memory stores, queue adapters, MCP client wiring
Endpoints/    # Minimal API route registrations (counter only)
```

### Microsoft Agent Framework

- **Package**: `Microsoft.Agents.AI` (NuGet) — part of `microsoft/agent-framework`
- **Role**: Provides the agent runtime, tool-calling abstractions, and multi-agent
  orchestration primitives used by the **counter**, **barista**, and **kitchen**
  in-process components.
- **LLM Provider**: The GitHub Copilot SDK (`CopilotClient`) IS the LLM provider
  for all MAF agents. Agents are created via the `copilotClient.AsAIAgent()`
  extension method — no Azure OpenAI credentials or GitHub Models endpoint
  configuration is required. Authentication is handled transparently by the
  GitHub Copilot CLI.
- **Key integration pattern** (from `Agent_With_GitHubCopilot` sample):
  ```csharp
  using GitHub.Copilot.SDK;
  using Microsoft.Agents.AI;

  // CopilotClient is the singleton provider (started once at startup)
  await using CopilotClient copilotClient = new();
  await copilotClient.StartAsync();

  // Create a MAF AIAgent backed by Copilot SDK
  SessionConfig sessionConfig = new()
  {
      Model     = "claude-opus-4.5",  // or any supported model
      Streaming = false               // set true for streaming responses
  };

  AIAgent agent = copilotClient.AsAIAgent(
      sessionConfig,
      ownsClient:  false,             // false when client is a shared singleton
      id:          "counter-agent",
      name:        "Counter Agent",
      description: "Coffee shop counter orchestrating agent"
  );

  // Non-streaming run
  AgentResponse response = await agent.RunAsync(userPrompt);

  // Streaming run
  await foreach (AgentResponseUpdate update in agent.RunStreamingAsync(userPrompt))
      Console.Write(update.Text);
  ```
- **Rules**:
  - Agent instructions MUST be defined as string constants in the component's
    `Application/` layer — no inline prompts in route handlers.
  - Tool implementations MUST reside in `Application/` (or `Infrastructure/` for
    I/O-bound tools); they MUST be pure functions with no hidden side-effects.
  - The MAF agent loop (`RunAsync` / `RunStreamingAsync`) is the sole mechanism
    for calling LLM APIs; raw `HttpClient` calls to LLM endpoints are NOT
    permitted.
  - `ownsClient: false` MUST be used when the `CopilotClient` is a DI singleton
    shared across agents; `ownsClient: true` is only permitted for short-lived
    standalone client instances.

### GitHub Copilot SDK

- **Package**: `GitHub.Copilot.SDK` (NuGet) — `github/copilot-sdk`
- **Scope**: Available to the **counter** component only.
- **Role**: Powers the **chat client** inside the counter — the conversational
  turn handler that receives user prompts via the CopilotKit AG-UI endpoint and
  manages a `CopilotSession` per user turn. This is distinct from the MAF agent
  loop; the Copilot SDK handles the interactive chat lifecycle while MAF tools
  perform domain actions (lookup, order creation, etc.).
- **Cookbook reference**: All implementation patterns MUST follow the official
  cookbook at `github/awesome-copilot/cookbook/copilot-sdk/dotnet`.
- **Key API pattern** (from cookbook):
  ```csharp
  // Singleton client — registered once in DI, started at startup
  await using var client = new CopilotClient();
  await client.StartAsync();

  // Per-turn session with stable custom ID (one per user / conversation)
  var session = await client.CreateSessionAsync(new SessionConfig
  {
      SessionId = $"counter-{customerId}",   // meaningful, trackable ID
      Model     = "gpt-4o",
      Provider  = new ProviderConfig          // GitHub Models
      {
          Type    = "openai",
          BaseUrl = "https://models.inference.ai.azure.com",
          ApiKey  = Environment.GetEnvironmentVariable("GITHUB_TOKEN")!
      }
  });

  var done = new TaskCompletionSource<string>();
  session.On(evt =>
  {
      if (evt is AssistantMessageEvent msg) done.SetResult(msg.Data.Content);
      if (evt is SessionErrorEvent err)     done.SetException(new Exception(err.Data.Message));
  });

  await session.SendAsync(new MessageOptions { Prompt = userPrompt });

  // Always await completion before responding to client
  using var cts = new CancellationTokenSource(TimeSpan.FromSeconds(30));
  var response = await done.Task.WaitAsync(cts.Token);

  await session.DisposeAsync(); // dispose after every turn
  ```
- **Rules** (derived from cookbook):
  - `CopilotClient` MUST be registered as a **singleton** in the DI container
    and started once at application startup (`StartAsync()`). Use `await using`
    or explicit `StopAsync()` in the application shutdown hook for cleanup.
  - Sessions MUST use a **meaningful `SessionId`** (e.g., `counter-{customerId}`)
    to enable session persistence and resumption across requests
    (`ResumeSessionAsync`).
  - Each session MUST be **disposed after the turn completes** (`DisposeAsync()`);
    sessions MUST NOT outlive a single user interaction turn.
  - **Error handling** MUST follow the cookbook `error-handling` recipe:
    - Catch `FileNotFoundException` — Copilot CLI not installed.
    - Catch `HttpRequestException` — CLI connectivity failure.
    - Always wrap long-running `done.Task` awaits with a `CancellationTokenSource`
      timeout (30 seconds default); surface `OperationCanceledException` as a
      user-facing timeout message.
    - Register a `Console.CancelKeyPress` / `IHostApplicationLifetime` shutdown
      hook that calls `client.StopAsync()` for graceful cleanup.
  - The `GITHUB_TOKEN` MUST be supplied via `dotnet user-secrets` locally and via
    environment variable in CI/production; it MUST NOT be hard-coded.

### Backend

- **Runtime**: .NET 10
- **Style**: Minimal API (no MVC controllers)
- **Language**: C# 13
- **Agent framework**: Microsoft Agent Framework (`Microsoft.Agents.AI`)
- **Architecture**: Clean Architecture (Domain → Application → Infrastructure → Endpoints)
- **Components**: counter (in-process, HTTP entry point), barista (in-process),
  kitchen (in-process), product-catalog (out-process MCP Server — separate project)
- **Unit testing**: xUnit + WebApplicationFactory (intra-service HTTP tests only)
- **Integration testing**: xUnit + `Aspire.Hosting.Testing` (see Integration Testing section)
- **Serialization**: System.Text.Json (built-in; Newtonsoft NOT permitted)
- **Storage**: In-memory only (see Storage section; no EF Core, no database)
- **Aspire integration**: all components and the frontend are registered in AppHost:
  ```csharp
  // Out-process MCP Server — declared first so counter can reference it
  var productCatalog = builder
      .AddProject<Projects.ProductCatalog>("product-catalog");

  // Counter: HTTP entry point; receives product-catalog URL via service discovery
  var counter = builder
      .AddProject<Projects.Counter>("counter")
      .WithReference(productCatalog);

  // In-process workers: receive internal queue reference from counter
  builder.AddProject<Projects.Barista>("barista")
      .WithReference(counter);

  builder.AddProject<Projects.Kitchen>("kitchen")
      .WithReference(counter);

  // Frontend: Next.js dev server; counter URL + OTLP endpoint injected as env vars
  builder.AddNpmApp("frontend", "../frontend", "dev")
      .WithReference(counter)
      .WithHttpEndpoint(env: "PORT");
  ```
