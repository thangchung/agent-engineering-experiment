## Ambiguities & Conflicts to Resolve

- [x] CHK051 — **[Conflict]** The spec states "order ready — all items complete" as a streaming message (FR-020d) and "ready" as an `OrderStatus` enum value. Are these the same event, or does the AG-UI stream "ready" before the HTTP response reflects the `OrderStatus` change? This needs a single defined answer. [Conflict, Spec §FR-020, §Key Entities — Order Status] — ✅ Resolved: spec.md §Session 2026-02-23 — atomic: counter emits AG-UI event and writes OrderStatus=Ready in the same operation; GET after stream event MUST return `status: "ready"`.
- [x] CHK052 — **[Ambiguity]** FR-009 allows item changes "before confirming" while FR-010 allows note changes "before `preparing`". Are these two different mutability windows intended (pre-confirm vs. pre-prepare), or should FR-009 also apply up to `preparing`? The PATCH contract should reflect whichever is correct. [Ambiguity, Spec §FR-009, FR-010, FR-011] — ✅ Resolved: spec.md §Session 2026-02-23 — FR-009 governs pre-confirmation cart; FR-010/011 govern PATCH (post-confirmation); both share the same boundary (before `preparing`); not in conflict.
- [x] CHK053 — **[Gap]** No requirement defines what happens when `Task.WhenAll` (parallel barista + kitchen dispatch) has one worker succeed and the other fail. Is the order partially completed? Rolled back? This is an unaddressed exception flow. [Gap, Coverage — Exception Flow] — ✅ Resolved: plan.md §Channel<T> — Partial Task.WhenAll failure → full failure: order Cancelled, user-facing error, no partial state persisted.
- [x] CHK054 — **[Gap]** The spec requires the system to "never persist a partial order state" (SC-008/Edge Cases), but with in-memory `ConcurrentDictionary` and no transaction support, there is no atomicity guarantee. Is an explicit compensating requirement (e.g., only write to store after all validation passes) documented? [Gap, Spec §SC-008, Edge Cases] — ✅ Resolved: plan.md §Concurrency & Atomicity — Pre-placement validation sequence (6 steps): construct order in memory first, TryAdd only after ALL validation passes.
